# Project Documentation: Coffee Machine Portal

## Project Origins

I built this project to serve as a robust, two-tier approval system for managing a fleet of premium coffee machines. The goal was to create something that looked aesthetically pleasing (using a custom coffee color palette) while enforcing strict business logic on the backend for how items get added, edited, and deleted from an inventory.

## The Tech Stack

I chose a stack that balances speed of development with modern performance standards:

**Frontend: React, Vite, and Tailwind CSS**
I went with React packaged by Vite because it's currently the fastest way to get a snappy single-page application up and running. 
For styling, Tailwind CSS was the obvious choice. Instead of writing separate CSS files, I configured a custom `coffee` color palette in `tailwind.config.js` (`#4A3B32`, `#F3F2F1`, etc.) and used Tailwind utilities to build a responsive, mobile-first UI. I heavily relied on Flexbox and CSS Grid to ensure dashboards and comparison tables scaled down perfectly for mobile devices.

**Backend: FastAPI, Python, and SQLite**
I used FastAPI for the backend. It's incredibly fast to write, automatically generates interactive API documentation (Swagger UI), and handles data validation using Pydantic models.
For data storage, I used SQLite via the SQLAlchemy ORM. This allows the app to be self-contained without needing the user to set up a separate PostgreSQL or MySQL server, while still allowing us to write clean, secure database queries in Python.

## Architecture & Implementation Details

### The Database Model
The core of the app revolves around a single `Machine` database model. It stores standard details (name, brand, price, description) but critically includes a `status` column. This `status` column drives the entire application logic.

### State & Navigation
The frontend uses standard React `useState` and `useEffect` hooks to fetch data from the FastAPI backend using Axios. We use `react-router-dom` to handle page navigation, passing machine IDs in the URL parameters so the details pages know which machine to fetch.

### The Two-Level Approval Pipeline
The most complex part of the implementation is the role-based state machine. Here is how the roles work in practice:

#### 1. L1 Reviewer (The Operator)
L1 users are the data enterers. They cannot directly alter the live, approved inventory.
- **Adding:** When L1 adds a machine, its status becomes `New`.
- **Editing:** When L1 edits an `Approved` machine, the status changes to `Pending Level 2`. The changes are saved temporarily, waiting for L2.
- **Deleting:** L1 cannot delete `Approved` machines. They can only click "Request Deletion," which changes the status to `Pending Deletion`. The only time L1 can directly delete a machine is if it has been marked as `Rejected`.

#### 2. L2 Reviewer (The Approver)
L2 users cannot edit data directly; their job is purely to review the queue generated by L1.
- **Approving:** L2 sees items that are `New` or `Pending Level 2`. If they approve, the status changes to `Approved` and the machine goes live.
- **Rejecting:** If they reject, the status changes to `Rejected`. 
- **Handling Deletions:** If L1 requested a deletion (`Pending Deletion`), L2 can either Approve (which actually deletes the record from the database) or Reject (which reverts the status, saving the machine).

#### 3. Admin (Superuser)
The Admin role is a convenience override. They have the UI elements and permissions of both L1 and L2 simultaneously, allowing them to make edits and immediately approve their own changes if necessary.

## Design Philosophy

The UI was built with a "premium" feel in mind. I used lots of border-radius (`rounded-2xl`, `rounded-3xl`), soft drop shadows, and a warm color scheme. I also implemented a custom floating action bar for the "Compare" feature that sticks to the bottom of the screen, providing a native app-like experience on mobile devices.
